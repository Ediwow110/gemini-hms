Blocking Findings
1. High: /settings was broadened from permission-gated governance access to role-based Super Admin + Branch Admin access.
   Evidence:
   - hms-frontend/src/App.tsx:300-310
     path: 'settings'
     element: <PermissionRoute allowedRoles={['Super Admin', 'Branch Admin']}><SettingsLayout /></PermissionRoute>
   - This causes all child routes under /settings to inherit the broadened access:
     - /settings
     - /settings/branches
     - /settings/departments
     - /settings/services
     - /settings/numbering
     - /settings/templates
     - /settings/notifications
     - /settings/security
   Why this is a problem:
   - The prior route pattern for adjacent governance surfaces remains permission-based:
     - hms-frontend/src/App.tsx:295-298
       - admin/users/:id -> admin.role.change
       - admin/roles -> admin.role.change
       - admin/roles/:id -> admin.role.change
       - admin/catalog -> catalog.manage
   - The route metadata still says /settings requires admin.role.change:
     - hms-frontend/src/config/portalRoutes.ts:51
       { path: 'settings', requiredPermission: PERMISSIONS.ADMIN_ROLE_CHANGE, zone: 'staff' },
   Impact:
   - This is not just navigation cleanup. It is a real guard change on a global settings area.
   - The attached report did not acknowledge this as a policy decision or risk.

2. Medium: The claimed Receptionist permission fix is not runtime-proven.
   Evidence:
   - hms-frontend/src/config/permissions.ts:45-48 adds:
     'Receptionist': [
       PERMISSIONS.QUEUE_VIEW,
       PERMISSIONS.QUEUE_MANAGE,
     ],
   - Repo-wide search found no consumer of ROLE_DEFAULT_PERMISSIONS besides its declaration.
   Why this is a problem:
   - Adding a role map that is never read does not actually change logged-in user permissions from /v1/auth/me.
   - The attached report described this as a real access fix, which is not supported by current evidence.

3. Medium: Quick Create still overclaims queue mutation capability.
   Evidence:
   - hms-frontend/src/app/AppShell.tsx:33
     const canAdmitQueue = isSuperAdmin || hasPermission('queue.view');
   - hms-frontend/src/app/AppShell.tsx:247-248
     Label: "Admit Queue Ticket"
     Description: "Issue active patient queue slot"
   - hms-frontend/src/features/queue/Queue.tsx:85 and :94
     Both "Call Next" and "Add to Queue" are wrapped in <RequirePermission permission="queue.manage">
   Why this is a problem:
   - A user with queue.view but without queue.manage can still see a Quick Create action that sounds transactional.
   - The modal opens the queue page, but the wording implies the user can issue or admit a queue slot directly.
