3. EXECUTIVE SUMMARY
======================================================================

The app does not look heavy because of one obvious infinite loop or a broken React Query global policy.

The heaviness is coming from a combination of four confirmed structural causes:

1. Too many dashboards depend on chart-heavy analytics components.
2. Shared shell features are computed globally even when they are not role-critical.
3. Several dashboards combine multiple queries, multiple grids, and multiple derived transforms in a single first paint.
4. Some portal pages still use rich dashboard shells even when the role only needs a lean operational workspace.

The current React Query defaults are not the main problem:
- staleTime = 60_000
- retry = 1
- refetchOnWindowFocus = false

That means the "heavy feel" is more likely caused by page composition, analytics/chart cost, and shell-level overhead than by aggressive refetch churn.

Evidence:

Why it matters:

- Even when split into a vendor chunk, this still means chart pages pay a large parsing and execution cost once loaded.

Classification:

Required direction:
- reduce default chart usage on operational pages
- lazy-load chart sections inside dashboards where possible
- avoid chart components for pages whose main job is workflow execution, not analytics


4.2 HIGH - Shared shell contains global UX features that every portal pays for, even when the role does not need them

Evidence:

  - command palette mount

Why it matters:
- This shell wraps almost every protected route.
- Even when a role only needs a focused work queue, the user is still carrying a broad admin-like shell.
- This increases perceived complexity and interaction cost, even if raw CPU cost is moderate.

Classification:
- Confirmed UX/performance architecture issue

Required direction:
- trim shell behavior by role/portal

======================================================================
8. SHOULD CACHE BE ADDED NOW?
======================================================================

Short answer:
Not as the first move.

Why:
- The current "heavy feel" looks frontend-structural before it looks database-protection-driven.
- Adding cache too early could hide slow UI architecture without actually improving user experience enough.
- You already have some measurement and React Query caching in place; the immediate win is lighter first paint and leaner role portals.

Best sequencing:
1. lighten shell and dashboard composition
2. reduce chart/default analytics cost on operational portals
3. make workflow-first roles land on leaner pages
4. then measure real backend latency again
5. only after that decide whether Redis or server-side cache is justified

9. HIGHEST-VALUE OPTIMIZATION PLAN
======================================================================

Priority 1 - Make operational portals lean
- Receptionist, queue, pharmacy, nurse, cashier, field roles should default to work queues or task boards, not analytics-heavy overview pages

Priority 2 - Contain chart cost
- move chart-heavy components behind deeper routes or lazy section boundaries
- keep analytics dashboards for admin/oversight users, not every workflow user

Priority 3 - Slim the global shell
- role-scope Quick Create
- role-scope notifications
- avoid broad command behavior everywhere
- consider lighter shell variants for narrow portals

Priority 4 - Memoize expensive global navigation transforms
- command palette allowedItems / filteredItems
- any full navigation flattening done on open/render

Priority 5 - Measure before adding infrastructure cache
- capture real browser timing on login, first route, and route-to-route navigation
- capture backend timing for auth/me and top dashboard endpoints
