## Confirmed bugs

### BUG-1: Tracked runtime config ships known secrets and auth-hardening bypass
Severity: P0
Status: Confirmed
Files:
- `docker-compose.yml`
- `docker-compose.prod.yml`

Evidence:
- `docker-compose.yml` contains:
  - hard-coded `DATABASE_URL`
  - hard-coded `JWT_SECRET`
  - hard-coded `MASTER_MFA_KEY`
  - `DISABLE_AUTH_VERIFICATION: "true"`
- `docker-compose.prod.yml` still ships default prod DB fallbacks:
  - `${DB_USER:-hms_prod_user}`
  - `${DB_PASSWORD:-hms_secure_pass}`
  - `${DB_NAME:-gemini_hms_prod}`

Failure mode:
A tracked runnable environment uses known credentials and can disable MFA verification.

Impact:
Production-footgun: secret exposure and privileged auth weakening.

Reproduction idea:
Run `docker compose up` from tracked files and authenticate as a sensitive role.

Precise fix:
Remove hard-coded secrets/default prod creds from tracked compose files. Remove non-test MFA bypass flags. Fail closed if bypass flags appear outside test.

Regression test:
CI guard that fails on tracked `DISABLE_AUTH_VERIFICATION=true`, hard-coded JWT/MFA secrets, or default prod DB passwords.

### BUG-2: Refresh-token race returns a new plaintext refresh token whose hash was never stored
Severity: P1
Status: Confirmed
Files:
- `hms-backend/src/auth/session.service.ts`
- `hms-backend/src/auth/auth.service.ts`

Evidence:
- `SessionService.rotateRefreshToken()` returns existing session during 30s leeway when bcrypt compare fails.
- `AuthService.refreshTokens()` always generates `newRtPlain/newRtHash` and returns `generateTokenPair(..., newRtPlain)` regardless of whether DB hash rotated.

Failure mode:
Concurrent refresh can hand a client a brand-new refresh token that is not the persisted valid token.

Impact:
Session corruption, random logout, broken multi-tab refresh behavior.

Reproduction idea:
Login once, fire two refresh requests in parallel, keep the slower response token, then refresh again later.

Precise fix:
Return structured rotation result (`rotated` vs `replay-within-leeway`). Only return a new plaintext token if its hash was stored.

Regression test:
Concurrent refresh E2E proving no response returns an unsaved refresh token.

### BUG-3: Branch context is lost after refresh
Severity: P1
Status: Confirmed
Files:
- `hms-backend/prisma/schema.prisma`
- `hms-backend/src/auth/auth.service.ts`

Evidence:
- `Session` model has no `branchId`.
- `selectBranch()` issues branch-bound tokens.
- `refreshTokens()` regenerates tokens without branchId.
- `generateTokenPair()` only includes branchId when explicitly passed.

Failure mode:
After branch selection, access-token refresh drops branch context.

Impact:
Branch-bound flows break after refresh and other branch-isolation bugs become easier to hit.

Reproduction idea:
Login → select branch → call a branch-bound endpoint → refresh → call it again.

Precise fix:
Persist branch context in session or a validated related record and reissue it during refresh.

Regression test:
E2E asserting branchId survives refresh and branch-bound routes still succeed.

### BUG-4: Refresh endpoint trusts client cookie identity and is excluded from app CSRF policy
Severity: P1
Status: Confirmed
Files:
- `hms-backend/src/auth/auth.controller.ts`
- `hms-backend/src/auth/guards/csrf.guard.ts`

Evidence:
- `POST /api/v1/auth/refresh` is marked `@Public()`.
- It reads `refresh_token`, `session_id`, and `user_id` from cookies.
- Global `CsrfGuard` skips all `@Public()` handlers.

Failure mode:
Refresh is outside the server CSRF model and consumes client cookie identity hints instead of deriving identity from the validated session.

Impact:
Broken session-security invariant. `SameSite=Strict` reduces classic browser CSRF, but server trust boundaries are still wrong.

Reproduction idea:
Tamper `user_id` cookie and compare behavior against actual session ownership; verify refresh works without CSRF header while other unsafe endpoints do not.

Precise fix:
Use a dedicated refresh guard/path. Validate refresh by session ownership and stored hash only. Remove `user_id` cookie as an identity source.

Regression test:
Refresh without the proper refresh guard/CSRF policy fails; tampered `user_id` cookie has no effect.

### BUG-6: Encounter/EMR/clinical endpoints allow cross-branch access because branch enforcement is incomplete
Severity: P0
Status: Confirmed
Files:
- `hms-backend/src/encounters/encounters.controller.ts`
- `hms-backend/src/encounters/encounters.service.ts`
- `hms-backend/src/emr/encounter.controller.ts`

- `hms-backend/src/clinical/clinical.controller.ts`

- `hms-backend/src/clinical/clinical-note.service.ts`
- `hms-backend/src/clinical/prescription.service.ts`
- `hms-backend/src/clinical/referral.service.ts`
- `hms-backend/src/clinical/diagnosis.service.ts`

Evidence:
- `encounters.controller.ts` and `emr/encounter.controller.ts` use `@RequireBranchContext()` without `BranchGuard`.
- `clinical.controller.ts` uses `RolesGuard` only.
- Services use tenant-only lookups for branch-bound records:
  - encounter read/update paths
  - note create/update/lock
  - prescription get/cancel
  - referral get/status
  - diagnosis remove

Failure mode:
A branch-scoped user can fetch or mutate another branch’s encounter/clinical records by UUID or by omitting branch filters.

Impact:
Cross-branch clinical data breach and clinical record mutation.

Reproduction idea:
Create encounter/note/prescription/referral data in branch B and access it using a branch A user with same tenant permissions.

Precise fix:
Add `BranchGuard` to branch-bound controllers and enforce tenant+branch filters inside every service read/write.

Regression test:
Cross-branch negative tests for encounter/EMR/clinical fetch and mutation routes.

## Security and production invariants
- No tracked runtime config may contain real/default secrets or auth-bypass flags.
- Refresh identity must come from validated session ownership, not client cookie hints.
- Refresh rotation must be atomic and must never return an unsaved refresh token.
- Branch context must survive refresh.
- Every branch-scoped endpoint must enforce branch isolation in controller and service/query layers.
- Clinical and financial reads are not exempt from branch isolation.
- Claims, procurement, stock, billing, and lab mutations must not trust client-supplied branch, actor, amount, price, or status as source of truth.
- Stock adjustment requires adjustment-specific authorization, not ordinary dispense rights.
- Clinical/lab/stock/financial mutations and their audits must commit atomically.
- Versioned regulated workflows must use optimistic locking.
- Role names must come from one canonical vocabulary shared across seed, backend, frontend, and tests.
- Deployment workflows must be reproducible from repository code and explicit env injection.
