## Confirmed bugs

### BUG-1: Tracked runtime config ships known secrets and auth-hardening bypass
Severity: P0
Files: `docker-compose.yml`, `docker-compose.prod.yml`

Evidence:
- `docker-compose.yml` contains hard-coded `DATABASE_URL`, `JWT_SECRET`, `MASTER_MFA_KEY`, and `DISABLE_AUTH_VERIFICATION: "true"`.
- `docker-compose.prod.yml` defaults `DB_PASSWORD:-hms_secure_pass` and `DB_NAME:-gemini_hms_prod`.

Failure mode: Tracked runnable config exposes known credentials and can disable MFA.

Impact: Production-footgun. Secret exposure and auth hardening bypass from tracked files.

Precise fix: Remove hard-coded secrets from tracked compose files. Remove `DISABLE_AUTH_VERIFICATION` from non-test runtime config. Fail closed on missing envs.

Regression test: CI guard that fails on tracked `DISABLE_AUTH_VERIFICATION=true`, hard-coded JWT/MFA secrets, or default prod DB passwords.

### BUG-2: Refresh-token race returns unsaved plaintext token
Severity: P1
Files: `hms-backend/src/auth/session.service.ts`, `hms-backend/src/auth/auth.service.ts`

Evidence:
- `SessionService.rotateRefreshToken()` (session.service.ts:62-72): on bcrypt compare failure within 30s leeway, returns existing session without updating hash.
- `AuthService.refreshTokens()` (auth.service.ts:382-391): always generates `newRtPlain`/`newRtHash` and returns `generateTokenPair(..., newRtPlain)` — even when session was returned without hash rotation.

Failure mode: Concurrent refresh can hand a client a brand-new refresh token whose hash was never stored in the DB.

Impact: Session corruption, random logout, broken multi-tab refresh.

Precise fix: Return structured rotation result (`rotated` vs `replay-within-leeway`). Only return a new plaintext token if its hash was stored.

Regression test: Concurrent refresh E2E proving no response returns an unsaved refresh token.

### BUG-3: Branch context lost after refresh
Severity: P1
Files: `hms-backend/prisma/schema.prisma`, `hms-backend/src/auth/auth.service.ts`

Evidence:
- Session model (schema line 183+) has no `branchId`.
- `AuthService.selectBranch()` (auth.service.ts:312-340) calls `generateTokenPair(..., branchId)` which includes branchId in payload.
- `AuthService.refreshTokens()` (auth.service.ts:382-391) calls `generateTokenPair(user, roles, session.id, newRtPlain)` with no branchId argument.

Failure mode: After branch selection, access-token refresh drops branchId from the new token.

Impact: Branch-bound flows break after refresh. Other branch-isolation bugs become easier to exploit because the guard reads `user.branchId` from the token.

Precise fix: Persist `branchId` on Session or in a related table. Reissue it during refresh.

Regression test: E2E asserting branchId survives refresh and branch-bound routes still succeed.

### BUG-4: Refresh endpoint trusts client cookie identity and is excluded from app CSRF policy
Severity: P1
Files: `hms-backend/src/auth/auth.controller.ts`, `hms-backend/src/auth/guards/csrf.guard.ts`

Evidence:
- `POST /api/v1/auth/refresh` is marked `@Public()` (auth.controller.ts:156).
- It reads `refresh_token`, `session_id`, and `user_id` from cookies (auth.controller.ts:159-161).
- `CsrfGuard.canActivate()` (csrf.guard.ts:19) returns `true` for all `@Public()` handlers — refresh bypasses CSRF entirely.

Failure mode: Refresh is outside CSRF model and consumes client cookie identity hints (`user_id`) instead of deriving identity from validated session ownership.

Impact: Session-security invariant broken. `SameSite=Strict` reduces browser CSRF risk, but server trust boundaries are still wrong.

Precise fix: Use a dedicated refresh guard. Validate refresh by session ownership and stored hash only. Remove `user_id` cookie as identity source.

Regression test: Tampered `user_id` cookie has no effect on refresh outcome.

### BUG-5: MFA enforcement misses real privileged seeded roles
Severity: P1
Files: `hms-backend/src/auth/auth.service.ts`, `hms-backend/src/auth/guards/mfa.guard.ts`, `hms-backend/prisma/seed.ts`

Evidence:
- Hard-coded sensitive set: `['Super Admin', 'Branch Admin', 'Doctor', 'Cashier', 'HR', 'Finance']` (auth.service.ts:32-39, mfa.guard.ts:13-19).
- Seeded privileged roles: `HR Manager`, `IT Support`, `Compliance Officer` (seed.ts:220-222).
- `HR` (lowercase, singular) does not match `HR Manager`.

Failure mode: Users with `HR Manager`, `IT Support`, or `Compliance Officer` roles bypass MFA entirely.

Impact: Approval/audit/compliance workflows can run without MFA.

Precise fix: Drive MFA from permission/role risk metadata, not fragile role-name string matching.

Regression test: E2E auth tests for every privileged seeded role that should require MFA.

### BUG-6: Encounter/EMR/clinical endpoints allow cross-branch access
Severity: P0
Files: `encounters.controller.ts`, `encounters.service.ts`, `emr/encounter.controller.ts`, `emr/encounter.service.ts`, `clinical.controller.ts`, `clinical/encounter.service.ts`, `clinical-note.service.ts`, `prescription.service.ts`, `referral.service.ts`, `diagnosis.service.ts`

Evidence:
- `encounters.controller.ts` and `emr/encounter.controller.ts` use `@RequireBranchContext()` without `BranchGuard`. `clinical.controller.ts` uses `RolesGuard` only.
- Service lookups across these modules use `{ id, tenantId }` without branch:
  - `encounters.service.ts:findOne()` (line ~123)
  - `clinical/encounter.service.ts:getEncounter()` (line ~82)
  - `clinical-note.service.ts:createNote()` (line ~33) validates encounter by tenantId only
  - `prescription.service.ts:getPrescription()` (line ~96) tenant-only
  - `prescription.service.ts:cancelPrescription()` (line ~126) tenant-only
  - `referral.service.ts:getReferral()` (line ~96) tenant-only
  - `referral.service.ts:updateReferralStatus()` (line ~124) tenant-only
  - `diagnosis.service.ts:removeDiagnosis()` tenant-only

Failure mode: Branch-scoped users can fetch or mutate another branch's encounter/clinical records by UUID.

Impact: Cross-branch clinical data breach and mutation.

Precise fix: Add `BranchGuard` to controllers. Pass `branchId` into service methods and enforce tenant+branch in queries.

Regression test: Cross-branch negative tests for every encounter/EMR/clinical fetch and mutation.

### BUG-7: Prescription query leaks across branches
Severity: P0

Impact: Cross-branch medication history exposure.

Precise fix: Pass `branchId` into `findByPatient()` and filter by branch unless explicit tenant-wide policy.

Regression test: Branch A user only receives branch A prescriptions.

### BUG-9: Pharmacy dispense and stock deduction do not share one transaction
Severity: P0
Files: `pharmacy.service.ts`, `inventory.service.ts`

Evidence:
- `pharmacy.service.ts:206-268` opens a Prisma transaction (`this.prisma.$transaction(async (tx) => { ... })`).
- Inside it at line 216, `this.inventoryService.dispenseItem(...)` is called.
- `inventory.service.ts:296-383` opens its own `this.prisma.$transaction(...)`.

Failure mode: Inventory deduction commits even if outer prescription transaction (audit, post-audit work) fails.

Impact: Prescription status and stock can diverge.

Precise fix: Accept optional transaction client in inventory methods. Use single transaction boundary.

Regression test: Forced outer failure after stock deduction must roll back both prescription and stock.

### BUG-10: Manual inventory dispense allows negative quantity
Severity: P0
Files: `inventory.controller.ts`, `inventory.service.ts`

Evidence:
- Controller (inventory.controller.ts:106-115): `@Body('quantity') quantity: number` — primitive, no validation.
- Service (inventory.service.ts:289-290): checks `if (stock.quantity < quantity)`, then computes `newStock = previousStock - quantity`.

Failure mode: `quantity = -5` passes `stock.quantity < -5` (assuming stock > 0) and increases stock.

Impact: Direct stock corruption via public API.

Precise fix: Use validated DTO with `@IsNumber() @Min(1) quantity` and re-verify server-side.

Regression test: Negative/zero/non-numeric quantity all fail with 400 and no stock change.

### BUG-11: Low-stock alert dedupe suppresses cross-branch alerts

### BUG-13: LabResult.version exists but lab mutations do not use optimistic locking
Severity: P1

## Security and production invariants
1. No tracked config may contain real/default secrets or auth-bypass flags.
2. Refresh identity must come from validated session ownership, not client cookie hints.
3. Refresh rotation must be atomic — never return an unsaved RT.
4. Branch context must survive refresh.
5. Every branch-scoped endpoint must enforce branch isolation in both controller and service layers.
6. Clinical and financial reads are not exempt from branch isolation.
7. Claims, procurement, stock, billing, and lab mutations must not trust client-supplied branch, actor, amount, price, or status as source of truth.
8. Stock adjustment requires dedicated authorization, not dispense permission.
9. Clinical/lab/stock/financial mutations and their audits must commit atomically.
10. Versioned regulated workflows must use optimistic locking.
11. Role names must come from one canonical vocabulary shared across seed, backend, frontend, and tests.
12. Deployment workflows must be reproducible from repository code and explicit env injection.
